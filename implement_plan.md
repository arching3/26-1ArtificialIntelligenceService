# DART 청크 및 요약 카드 품질 개선 구현 계획

작성일: 2026-06-09

## 1. 목적

현재 DART 정기공시 청크는 XML 구조를 일부 반영하고 있지만, 제목 오인식과 잘못된 섹션 계층으로 인해 사업 부문이 섞이거나 특정 문서 후반부의 내용이 요약 근거를 과도하게 점유할 수 있다.

특히 요약 서비스는 최신 `business_text` 청크 3개를 `updated_at DESC, id DESC` 순서로 선택한다. 같은 시점에 적재된 청크에서는 사실상 문서 마지막 청크가 선택되므로 기업 전체가 아닌 특정 제품, 자회사 또는 사업 부문 중심의 요약이 생성될 수 있다.

이번 작업은 다음 두 단계로 진행한다.

1. 문서 구조 분석과 청크 생성 품질 개선
2. 임베딩, 검색, evidence 선별 및 요약 검증 개선

핵심 품질 기준은 문장이 자연스러운지가 아니라 다음과 같다.

- 청크의 섹션과 사업 부문 메타데이터가 원문 구조와 일치하는가
- 카드별 evidence가 해당 카드의 목적과 일치하는가
- 선택한 evidence가 기업 전체를 충분히 대표하는가
- 재무 수치와 주요 주장이 제공된 근거로 검증 가능한가

## 2. 현재 확인된 문제

### 2.1 제목 오인식

`backend/src/document_processor.py`의 `_looks_like_heading()`은 짧은 문장에 `생산`, `매출`, `투자` 같은 키워드가 포함되면 제목으로 판단할 수 있다.

이 때문에 다음과 같은 일반 문장이 제목 또는 섹션 경로로 저장된다.

```text
해외에서는 생산, 판매, 연구활동 등을 담당하는 종속기업이 운영되고 있습니다.
Harman의 2026년 1분기 가동률은 생산능력 대비 생산실적으로 산출하였습니다.
```

### 2.2 사업 부문 경계 누락

`[DX 부문]`, `[DS 부문]`, `[SDC]`, `[Harman]`과 같은 대괄호 형태의 부문 전환이 독립적인 구조 경계로 처리되지 않는다.

그 결과 서로 다른 사업 부문의 문단이 같은 청크 또는 같은 `section_path` 아래에 포함될 수 있다.

### 2.3 잘못된 섹션 계층 전파

제목 레벨이 번호 체계보다 정규식과 키워드에 의존한다. 잘못 인식된 제목 하나가 이후 모든 하위 청크의 `section_path`에 남을 수 있다.

### 2.4 표 청크 과다

현재 표 하나를 독립 청크 하나로 저장한다. 삼성전자 기준 전체 503개 청크 중 315개가 `table_text`이며, 전체의 약 62.6%다.

작은 표도 독립 벡터가 되므로 검색 결과를 과도하게 점유하고, 표 제목과 단위 또는 인접 설명이 부족한 상태로 반환될 수 있다.

### 2.5 요약 evidence 선택 편향

`SummaryService`는 카드 요약에 최신 `business_text` 3개, `risk_text` 2개, `event_text` 3개만 사용한다.

이 선택은 검색 결과가 아니라 문서 저장 순서에 가깝다. 문서 후반부에 특정 자회사나 제품 설명이 있으면 해당 내용이 회사 전체 설명처럼 요약될 수 있다.

## 3. 제목 판별 개선

### 3.1 대상 파일

- `backend/src/document_processor.py`
- `tests/check_chunk_structure.py`

### 3.2 구현 내용

제목 판별 우선순위를 다음과 같이 변경한다.

1. 실제 XML 또는 HTML 제목 태그
2. DART 원문의 제목 관련 속성 또는 스타일
3. 명확한 번호 체계
4. 대괄호 형태의 사업 부문 마커
5. 제한적인 heuristic

단순 키워드 포함만으로 제목을 판정하는 현재 동작은 제거한다.

다음 조건에 해당하는 텍스트는 제목 후보에서 제외한다.

- `입니다`, `합니다`, `있습니다`, `되었습니다` 등 종결형 문장
- 숫자와 설명이 결합된 완결 문장
- 기본 60자를 초과하는 문장
- 마침표로 끝나는 긴 설명문
- 표의 단위, 출처 또는 주석

제목 판별 결과에는 다음 정보를 추가한다.

```python
{
    "heading_source": "tag|number_pattern|segment_marker|heuristic",
    "heading_confidence": 0.0,
}
```

### 3.3 제목 번호 체계

별도 함수 `_parse_heading_number()`를 추가하고 다음 패턴을 우선 처리한다.

```text
II. / Ⅱ. / 제2부       level 1
1. / 2.                 level 2
가. / 나.               level 3
1) / (1)                level 4
가) / (가)              level 5
```

명시적인 태그 레벨이 있으면 번호 체계보다 우선할 수 있지만, 두 정보가 충돌하면 로그와 메타데이터에 남긴다.

### 3.4 완료 조건

- 일반 설명 문장이 제목으로 분류되지 않는다.
- `1. 사업의 개요`, `가. 주요 사업`, `(1) 제품별 현황`은 제목으로 인식된다.
- `사업의 내용` 시작과 `재무에 관한 사항` 종료 경계를 유지한다.
- 제목 판별 결과를 단위 테스트로 재현할 수 있다.

## 4. 사업 부문 경계 추출

### 4.1 대상

- `ParsedBlock`
- `ParsedSection`
- `BusinessChunk`
- `parse_filing_document()`
- `extract_business_sections()`

### 4.2 신규 함수

```python
def _detect_segment_marker(text: str) -> SegmentMarker | None:
    ...
```

인식 대상 예시는 다음과 같다.

```text
[사업부문]
[연결회사]
[주요 종속회사]
[회사명]
가. 반도체 부문

```

특정 기업의 사업 부문명은 하드코딩하지 않는다.

다음 표현은 사업 부문 마커에서 제외한다.

```text
[단위: 백만원]
[기준일: 2026.03.31]
[주]
[참고]
```

### 4.3 문단 내부 경계 처리

한 블록 안에 사업 부문 마커가 포함된 경우 해당 위치에서 블록을 분리한다.

```text
기존 사업 설명...

[다른 부문]

새 사업 설명...
```

분리된 이후의 청크에는 새로운 `segment_name`을 적용한다.

### 4.4 메타데이터

```python
{
    "segment_name": "원문 사업 부문명",
    "segment_path": ["상위 사업", "하위 사업"],
    "segment_source": "bracket|heading|table|inferred",
}
```

원문 사업 부문명은 보존하고, 비교와 중복 제거에 사용할 정규화 값은 별도로 생성한다.

### 4.5 완료 조건

- 서로 다른 사업 부문의 본문이 하나의 청크에 혼합되지 않는다.
- 사업 부문이 없는 기업도 기존처럼 정상 처리된다.
- 제조업, 금융업, 플랫폼, 바이오, 지주사에 같은 규칙을 적용할 수 있다.

## 5. 섹션 계층 안정화

### 5.1 구현 내용

`extract_business_sections()`의 heading stack 관리를 다음과 같이 변경한다.

1. 동일 레벨 또는 상위 레벨 제목이 나오면 기존 하위 경로를 닫는다.
2. 새로운 대단원이 나오면 이전 하위 경로를 초기화한다.
3. 레벨이 비정상적으로 건너뛰면 최근 유효 상위 제목 아래에 배치한다.
4. 동일 제목이 연속 반복되면 하나로 병합한다.
5. 긴 완결 문장은 경로에 포함하지 않는다.
6. 사업 부문 경계는 일반 제목 계층과 별도 축으로 관리한다.

예상 메타데이터:

```json
{
  "section_path": [
    "II. 사업의 내용",
    "1. 사업의 개요",
    "가. 주요 사업"
  ],
  "section_level": 3,
  "segment_name": "반도체 부문"
}
```

### 5.2 구조 이상 탐지

다음 상태를 검증 결과에 포함한다.

- 상위 제목 없이 지나치게 깊은 레벨이 등장
- 섹션 제목이 완결 문장 형태
- 동일 제목이 과도하게 반복
- 하나의 섹션 아래 서로 다른 사업 부문이 혼재
- `section_path`가 원문 순서와 충돌

### 5.3 완료 조건

- 반도체, 디스플레이, 모바일, 전장 등 서로 다른 내용이 잘못된 단일 하위 섹션에 포함되지 않는다.
- 일반 문장이 `section_path`에 저장되지 않는다.
- 원문 순서와 섹션 순서를 비교할 수 있다.

## 6. 표 문맥 결합

### 6.1 대상 파일

- `backend/src/document_processor.py`
- `tests/check_table_chunks.py`

### 6.2 구현 내용

표 청크 생성 시 다음 문맥을 수집한다.

- 가장 가까운 유효 제목
- 표 직전의 설명 문단
- 표 제목
- 단위 표기
- 필요한 경우 표 직후의 해설 문단

표 청크의 표시용 본문은 다음 형태로 구성한다.

```text
섹션: 4. 매출 및 수주상황
사업 부문: 반도체
표 제목: 사업부문별 매출
단위: 백만원
설명: 당사의 부문별 매출 현황은 다음과 같습니다.

| 사업부문 | 당기 | 전기 |
| --- | --- | --- |
| ... | ... | ... |
```

인접 문맥은 무제한으로 포함하지 않는다.

- `context_before`: 최대 300~500자
- `context_after`: 표를 직접 설명하는 경우에만 최대 300자

### 6.3 메타데이터

```python
{
    "table_caption": "",
    "table_unit": "",
    "context_before": "",
    "context_after": "",
    "row_count": 0,
    "col_count": 0,
}
```

### 6.4 완료 조건

- 표 청크만 조회해도 표의 의미와 단위를 파악할 수 있다.
- 무관한 인접 문단이 표 문맥으로 들어가지 않는다.
- 표 제목과 단위를 별도 메타데이터로 검색할 수 있다.

## 7. 작은 표 병합과 큰 표 분할

### 7.1 신규 함수

```python
def merge_or_split_tables(
    tables: list[ParsedBlock],
    *,
    max_chars: int,
    max_rows: int,
) -> list[BusinessChunk]:
    ...
```

### 7.2 작은 표 병합 기준

초기 기준:

```text
row_count <= 4
문자 수 <= 500
같은 section_path
같은 segment_name
원문에서 연속된 표
```

추가 조건:

- 표 제목 또는 단위가 동일하거나 호환되어야 한다.
- 병합 결과는 약 1,800자를 넘지 않는다.
- 표 사이에 중요한 설명 문단이 있으면 병합하지 않는다.

### 7.3 큰 표 분할 기준

다음 중 하나에 해당하면 행 단위로 분할한다.

- 2,500자 초과
- 25행 초과
- 임베딩 모델 입력 또는 검색 품질에 부적절하게 큰 표

분할 규칙:

- 첫 번째 헤더 행을 모든 청크에 반복한다.
- 행 내부를 문자 단위로 자르지 않는다.
- 표 제목, 단위, 섹션, 사업 부문을 모든 청크에 유지한다.

```python
{
    "table_group_id": "",
    "table_part": 1,
    "table_part_total": 3,
}
```

### 7.4 제거 대상

- 헤더만 존재하는 표
- 모든 셀이 비어 있는 표
- 완전히 동일한 중복 표
- 단위나 주석 한 줄만 포함된 표

### 7.5 완료 조건

- 200자 미만 청크 비율이 기존보다 감소한다.
- 작은 표가 검색 결과를 과도하게 점유하지 않는다.
- 큰 표 분할 후 모든 부분에 헤더가 남는다.
- 행 데이터가 중간에서 잘리지 않는다.

# 2단계: 임베딩, 검색, evidence 및 요약 개선

## 10. 임베딩 텍스트 분리

### 10.1 목적

현재 `content`에는 사용자에게 필요한 원문과 검색에 불필요한 반복 헤더가 함께 포함된다.

다음 두 필드를 분리한다.

```python
{
    "content": "사용자 표시 및 출처 확인용 원문",
    "embedding_text": "검색과 임베딩에 최적화한 텍스트",
}
```

### 10.2 `embedding_text` 구성

포함:

- 유효한 섹션 경로
- 사업 부문
- 표 제목과 단위
- 실제 본문

제외:

- 기업코드
- 접수번호
- 청크 번호
- 반복 라벨
- 고정 안내 문구

예시:

```text
사업의 내용 > 사업의 개요
사업 부문: 반도체
주요 제품과 서비스

실제 본문...
```

### 10.3 대상 파일

- `backend/src/document_processor.py`
- `backend/src/dart_service.py`
- `backend/src/finance_store.py`
- `backend/src/index_manager.py`

### 10.4 DB 변경

```sql
ALTER TABLE chunks ADD COLUMN embedding_text TEXT;
```

기존 데이터 호환:

```python
embedding_text = chunk.get("embedding_text") or chunk["content"]
```

### 10.5 완료 조건

- 화면과 출처 표시는 기존 `content`를 사용한다.
- FAISS 임베딩은 `embedding_text`를 우선 사용한다.
- 기존 청크도 인덱스 재구축 전까지 동작한다.

## 11. 카드별 evidence 검색기

### 11.1 신규 모듈

```text
backend/src/summary_evidence.py
```

### 11.2 카드별 검색 목적

```text
overview:
  기업의 주요 사업, 제품, 서비스, 고객, 시장

benefit:
  수익원, 판매 방식, 계약 방식, 고객군, 매출 구조

earnings:
  정형 재무 데이터와 비교 가능한 기간 데이터

risk:
  사업, 법률, 시장, 공급망, 신용, 유동성 위험

changing:
  최근 사업 구조 변화, 인수, 매각, 신사업, 조직 변화

status:
  공모, 증자, 사채, 증권 발행, 상장, 자금조달

anomaly:
  비정기 중요 사건, 일회성 사건, 예외적인 공시
```

### 11.3 검색 절차

카드별로 다음 과정을 수행한다.

1. 목적별 검색어 생성
2. 관련 index와 `data_type` 결정
3. FAISS 후보 20~40개 검색
4. 최신성 가중치 적용
5. 섹션과 사업 부문 적합도 계산
6. 다양성과 대표성 기준으로 재정렬
7. 최종 4~8개 evidence 선택

`get_latest_chunks(..., limit=3)`는 요약 evidence 경로에서 제거한다.

### 11.4 완료 조건

- overview가 문서 마지막 청크에 의존하지 않는다.
- 카드별로 서로 다른 evidence 집합을 사용할 수 있다.
- 선택된 evidence의 ID와 선택 점수를 확인할 수 있다.

## 12. 대표성 및 다양성 재정렬

### 12.1 후보 점수

초기 점수 모델:

```text
최종 점수 =
  벡터 유사도
  + 섹션 적합도
  + 최신 보고서 가중치
  + 회사 전체 설명 가중치
  + 수치 및 기간 명확성
  - 동일 섹션 중복 페널티
  - 동일 사업 부문 과다 선택 페널티
  - 짧거나 문맥이 부족한 청크 페널티
```

가중치는 상수로 분리하고 테스트 결과에 따라 조정한다.

### 12.2 선택 제약

- 동일 `section_path`에서 최대 2개
- 동일 `segment_name`에서 최대 2개
- overview는 가능한 경우 서로 다른 사업 부문을 포함
- 표 청크는 전체 evidence의 절반 이하
- 최신 분기보고서만으로 전체 구조를 설명하기 어려우면 최근 사업보고서도 포함
- 동일 내용의 중복 청크는 하나만 선택

### 12.3 기업 구조별 처리

- 단일 사업 기업에는 부문 다양성을 강제하지 않는다.
- 지주사는 주요 자회사와 투자 구조를 대표 축으로 사용한다.
- 금융사는 상품, 고객, 수익원, 위험 유형을 대표 축으로 사용한다.
- 연구개발 중심 기업은 파이프라인, 기술, 개발 단계, 자금조달을 대표 축으로 사용할 수 있다.

기업 유형은 회사명이나 업종 하드코딩이 아니라 evidence 구조로 판단한다.

### 12.4 완료 조건

- 특정 자회사 또는 제품이 전체 evidence를 독점하지 않는다.
- 단일 사업 기업에는 불필요한 다양성 패널티를 적용하지 않는다.
- 선택 결과와 탈락 사유를 로그 또는 테스트 출력으로 확인할 수 있다.

## 13. 구조화된 LLM 응답

LLM 응답을 단순 문자열 딕셔너리에서 다음 형태로 확장한다.

```json
{
  "overview": {
    "text": "...",
    "evidence_ids": [101, 205],
    "coverage": ["business", "product", "market"]
  },
  "benefit": {
    "text": "...",
    "evidence_ids": [310, 311],
    "coverage": ["revenue_source", "customer", "sales_method"]
  }
}
```

프론트엔드 API는 호환성을 위해 기존 7개 문자열 필드를 계속 반환할 수 있다. 구조화된 상세 정보는 내부 저장 또는 별도 디버그 응답에 보관한다.

## 14. 요약 결과 검증

### 14.1 신규 모듈

```text
backend/src/summary_validator.py
```

### 14.2 검증 항목

1. 모든 `evidence_ids`가 실제 제공된 evidence인지 확인
2. 재무 숫자가 `financial_data`와 일치하는지 확인
3. evidence에 없는 숫자와 고유명사 검출
4. 카드 간 문장 또는 핵심 사실 중복 검사
5. `changing`과 `anomaly`가 같은 사건만 반복하는지 검사
6. overview와 benefit의 coverage 검사
7. 특정 사업 부문의 과도한 편중 검사
8. status가 시스템 상태가 아닌 공모 및 자금조달 상태인지 검사

### 14.3 재생성 정책

```text
첫 생성
→ validator 실행
→ 실패 이유와 함께 한 번 재생성
→ 다시 실패하면 통과한 카드만 유지
→ 실패한 카드만 deterministic fallback 적용
```

전체 요약 실패로 처리하지 않고 카드 단위로 복구한다.

### 14.4 완료 조건

- 근거 ID가 없는 주장을 탐지한다.
- 재무 수치 불일치를 탐지한다.
- 카드 간 중복이 기준을 넘으면 재생성한다.
- 재생성 실패 시 안정적으로 fallback한다.

## 19. 최종 승인 기준

### 청크 구조

- 일반 설명 문장이 제목 또는 섹션 경로로 저장되지 않는다.
- 서로 다른 사업 부문이 하나의 청크에 불필요하게 혼합되지 않는다.
- 표 제목, 단위와 행 헤더가 보존된다.
- 작은 표와 짧은 청크가 검색 결과를 과도하게 점유하지 않는다.

### 검색

- 카드별 evidence가 문서 저장 순서가 아니라 검색 목적에 따라 선택된다.
- 특정 제품이나 자회사만으로 회사 전체를 설명하지 않는다.
- 기업 구조에 따라 다양성 기준을 유연하게 적용한다.

### 요약

- 모든 핵심 주장과 숫자를 evidence로 추적할 수 있다.
- overview, benefit, earnings가 서로 다른 질문에 답한다.
- changing과 anomaly가 같은 내용을 반복하지 않는다.
- status는 공모, 증권 발행, 상장 또는 자금조달 상태만 설명한다.
- validator 실패 시 재생성과 fallback이 안정적으로 동작한다.

### 회귀 방지

- 기존 재무 추출, 이벤트 수집, 챗봇 검색과 frontend API가 유지된다.
- 검증 스크립트는 모두 프로젝트의 `tests/` 아래에 둔다.
- 변경된 청크 구조와 요약 캐시는 버전으로 무효화할 수 있다.
